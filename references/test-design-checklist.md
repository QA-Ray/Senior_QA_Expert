# 分层测试设计 · 硬核场景清单(通用层)

设计用例前过一遍。每一层给的是**逼问的角度**,不是要把每条都写成用例——而是确保不漏维度。挑与当前需求相关的深挖。
面向后端 / 资金 / 数据类系统;每一条都要问"这里出事会怎样"。

> **域专属清单**:各业务域(钱包充提、合约、现货、风控、签名…)**额外**要对的坑,见 `domains/<域>.md`(已沉淀 `domains/wallet.md`)。本文件是**任何域都适用的通用分层层**。

## 1. 核心路径
- 主流程从头到尾跑通,关键字段口径与需求/接口文档一致。
- 多入口能否到达同一状态(App / 后台 / 定时任务 / 补偿任务)。
- 一句话验收:这个功能"做对了"到底以什么为准?判据写死,别含糊。

## 2. 边界值与等价类
- 数值:0、1、最小、最大、最大+1、负数、超长、超精度(小数位)。
- 字符串:空、null、超长、特殊字符/emoji、SQL/脚本注入字符、前后空格。
- 集合:空列表、单元素、上限条数、超上限、分页最后一页/翻过尾页。
- 时间:闰年、月末、跨年、时区边界、夏令时、未来时间/过去时间。

## 3. 异常与错误处理
- 非法输入:类型错、格式错、枚举外的值、缺必填、多余字段。
- 资源态:不存在的 id/订单/资源、已删除、已冻结、已过期。
- 上游异常:超时、5xx、连接重置、限流、返回半截/脏数据。
- 错误是否给到明确错误码/文案,而不是 500 或静默吞掉?失败后状态是否可恢复?

## 4. 状态机与时序
- 列全状态与合法迁移,逐一试**非法迁移**(如 已完成→处理中、已取消→成功)。
- 重复触发同一动作(重复点提交、重复回调)结果是否一致。
- 乱序:后发生的事件先到(回调乱序);早期事件迟到。
- 超时后重试与原请求并存;补偿任务与正常流程撞车。

## 5. 并发与幂等
- 同一资源并发操作:并发下单/撤单、并发提交、并发改状态。
- 重复提交 / 网络重发:是否有幂等键?同一幂等键多次请求是否只生效一次。
- 资损类:双花、超卖、余额扣两次、奖励/红包领两次、手续费收两次。
- 分布式锁/乐观锁是否覆盖?锁失效或超时时的行为。
- 先扣后加 vs 先加后扣的中途崩溃;TCC/对账补偿是否兜底。

## 6. 接口 / 回调 / 集成
- 回调:乱序、重放(同一回调发多次)、丢失(回调没来,靠轮询/对账兜底)、伪造回调、回调验签失败。
- 异步:发起成功但回调迟迟不来,前端/账务状态停在哪。
- 部分成功:批量操作里一部分成功一部分失败,整体如何呈现与回滚。
- 上下游字段口径(单位、精度、枚举值、时区)是否一致;确认/回执的门槛。

## 7. 弱网与中断
- 弱网/高延迟下重复点击、loading 卡死、超时重试。
- 请求发出后断网:服务端已处理但客户端没收到响应 → 用户重试 → 是否重复。
- 操作中途杀进程/切后台/锁屏,恢复后状态是否一致。
- 接口慢导致的前端假死、按钮可重复点击。

## 8. 数据一致性与对账
- **读的是实时数据还是聚合/快照?** 很多系统对外读的是异步同步出来的聚合表,不是实时主库——同步延迟时会与实时库对不上,先确认这是"预期延迟"还是 bug。
- 多数据源口径对齐;缓存与 DB 不一致;最终一致的延迟窗口。
- **"总额"与"分项加和"是否对得上**(常见设计是不同口径/不同时间窗,可能本就对不上——先确认口径再下结论)。
- 对账差异:应入未入、已入重复、漏记、重复扣。
- 能用只读查询的,直接查实际数据核对,别凭文档想当然。

## 9. 金额 / 精度 / 费用(最易漏扣)
- **截断 vs 四舍五入**:不同系统对小数位的处理不同,算错预期值就会误判——先搞清这个币种/字段的取整规则。
- 单位与换算:本位币 / 法币 / 折算;后端常以字符串返回避免浮点误差。
- 精度累积:多笔累加后的尾差;手续费、利率、汇率换算的取整方向。
- 手续费/返佣/奖励:是否多扣、少扣、重复扣、负数、为 0 时的处理。
- 极端金额:0、极小(最小精度)、极大(溢出/越界)。

## 10. 性能与容量
- 大数据量:大账户海量记录、深翻页、导出大文件。
- 慢查询:无索引字段过滤、跨大表 join;聚合任务在数据量大时的耗时。
- 限流与降级:触发限流后的提示与重试;降级时返回的兜底数据是否标注。
- 定时任务积压、补数据对在线读的影响。

## 11. 安全与权限
- 鉴权:未登录、token 过期/伪造、越权访问他人数据(水平越权)。
- 垂直越权:普通账号访问管理接口/操作。
- 注入:SQL / 命令 / 脚本;参数篡改(改金额、改 id、改币种)。
- 重放:抓包重放下单/提现/支付/签名请求。
- 敏感信息:接口/日志/导出文件是否泄露密钥、手机号、身份、连接串、内部地址。
- 高风险资金链路:上线前先**小额闭环**跑通(如充值→提现),验证 OK 再放大额——防签名/链路缺陷导致大额取不回。

## 12. 兼容性与灰度
- 版本兼容:新老客户端、协议字段增删的向后兼容。
- 多端一致:App / Web / 后台对同一数据的展示口径。
- 灰度/AB:灰度名单内外行为差异、灰度回滚。

## 13. 隐性逻辑 / 文档口径
- **文档说的库/表/字段/枚举与实际实现是否一致**(逻辑名 ≠ 真实名、文档滞后是高频坑)。
- 默认值、空态、零态(没有数据时显示 0 还是 "-" 还是隐藏)。
- 时区跨天:按 UTC 还是本地取日期,跨天那一刻是否串日。
- 边缘业务规则:首单/首充优惠、临界达标、四舍五入后是否刚好够门槛。
- "看起来是 bug 其实是设计":遇到对不上的数,先去文档/代码确认口径,再下 Fail 结论。
