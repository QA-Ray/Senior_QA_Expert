# 域清单 · 钱包(充值 / 提现 / 归集 / 资产)

> 通用分层清单(`../test-design-checklist.md` §1-13,任何域都过)之外,钱包 / 充提类的**专属**坑。
> 设计这类用例时**逐项对**。充提类最容易"照需求文档写主干"就交差,恰恰漏掉下面这些。

## 0. 永远先分的两条主轴
- **主链币(原生币) vs 代币**:走**完全不同的链上路径**(原生转账 vs 合约调用),充、提、归集都要**各测一遍**,别用"币种无关"一条糊过去。
- **链特性按链分**(见 §7):同一个"提币",POL/TON/SOL/EVM 的失败模式天差地别。

## 1. 充值
- 主链币 / 代币 正常入账;确认数差 1 不入、达标才入(边界);高精度不丢精度。
- **共享地址 memo**:无 memo / memo 写错 → **挂账人工,不乱猜用户入账**(TON/EOS/XRP 高频丢币)。
- 不支持币种、低于最小充值额、**仿冒代币(假 USDT,合约地址对不上不被当真币)**、dust/0 金额、一笔交易多输出(给多用户)。
- 回调乱序 / 重投幂等;归属**按收款地址认人**(不信链上消息里的 uid);**区块重组(reorg)**:已侦测的充值被回滚不能误入账。
- 风控:充到高风险 / 黑名单地址的打标处理。

## 2. 提现
- 主链币 / 代币 正常出账;**代币转账需主链币付 gas → 自动代付 gas(手续费钱包/能量代理);gas 不足先补再发**。
- 失败全谱:入口拒收 / **余额不足(应转失败退款,别卡住)** / 签名失败(永不重试) / 广播失败(永久 vs 临时重试耗尽)。
- **上链了但执行 reverted(mined ≠ success)** → 不误标完成。
- **收款地址校验**:非法格式 / 合约地址 / **黑名单·AML 地址** / 本平台内部地址(走内部划转)。
- **nonce 卡单 → 加速/替换(RBF,EVM)**;全额提现(留 gas);幂等重投返首单;防伪前缀(提币禁 `sweep-`)。
- **风控限额**:单笔 / 单日限额、大额人工审核、提现冷却。
- 卡"广播中"超时:只告警、不擅自改状态。

## 3. 归集(易漏风控与经济性)
- 正常归集 → 热钱包(**不通知用户**、写完成态);同址去重;双源 uid 一致;归集也要 gas(代币归集的 gas 代付)。
- **风控:AML / 风控 / 黑名单地址禁止归集**(脏钱不混进热钱包,合规红线)。
- **经济性:余额低于最小归集额(归集 gas > 价值,如 0.1U)不归集**;按金额阈值 / 油价上限 / 价格波动选址。
- 批量上限(超限整批拒收,注意文档口径是否自相矛盾)。

## 4. 钱包资金管理(最容易整块漏)
- **冷热温内部调拨**(热→温 / 温→热 / 热→冷 / 冷→温);调拨的幂等与防伪,不与提币/归集单号冲突。
- **出款资产不足预警**:热钱包余额触告警线**提前告警**(不是等提现失败才暴露);手续费钱包余额告警。

## 5. 对账 / 偿付能力
- **链上实际余额 vs 账面**对账;**平台可控(冷热温) ≥ 用户负债**(不资不抵债);差异登记 / 告警。
- 资金流水只增不改、按唯一键(链:txHash:logIndex)幂等。

## 6. 安全
- 提现防篡改(verify 比对关键字段);**memo 不在防伪范围的缺口**(带 memo 链);越权(admin/biz 接口水平/垂直越权);地址投毒;敏感信息(私钥/助记词/内部地址)不从接口泄露;限流防刷。

## 7. 链特性(按链各测,别"链无关"糊)
- **EVM(POL 等)**:nonce 管理 / 卡单 / RBF、EIP-1559 gas 飙升、reorg 深度、**mined-but-reverted**、ERC20 approve/transfer。
- **SOL**:**SPL 代币账户(ATA)不存在时的处理**、**rent 豁免**、**blockhash 过期**、优先费、versioned tx。
- **TON**:共享地址需 memo、**jetton 转账需 TON 付 gas + 存储费**、bounce 退回、地址格式(bounceable / raw)。
- **通用**:入账确认数 vs 提现确认数分开、多节点共识、**RPC 节点故障切换**、**链高度落后告警**、扫块积压追块、签名后崩溃恢复。

## 8. 运维 / 配置 / 兼容
- 新链 / 新币种上线;**停用链 / 停用币种时在途充提怎么处理**;确认数配置变更;地址池补充(refill)。
- **老系统兼容**:老地址 V1/V2 的充提与归集(老地址签名路径)、`is_legacy_address` 处理、新老边界。

## 9. 多链复用:参数化设计(新增链 = 加一行,不重写)
钱包常支持多条链,每条链通常 1 个主链币 + 若干代币,且**充提归集的核心逻辑跨链一致**。要让用例和脚本可复用:
- **把"链 × 币种类型"做成参数矩阵**(一个 `chains.yaml` / 表)作为用例与脚本的**单一参数源**:每条链填 网络、原生币、代币(合约 / mint / jetton master + decimals)、入账 / 提币确认数、测试源钱包 / faucet、该链特有点(quirks)。
- **核心充提归集用例写成链无关模板**,对矩阵每个 (链 × {主链币, 代币}) 各跑一遍——同一套用例 + 同一套脚本,只换参数;**别为每条链复制一份用例**。
- **链特性用例(memo / ATA / jetton gas / nonce / blockhash / reorg…)按 quirk 单挂相应链**,不强加给别的链。
- **新增一条链 = 矩阵加一行**(填参数),用例与脚本都不改。脚本要**数据驱动**(读矩阵 + parametrize),不是每链一份拷贝。
